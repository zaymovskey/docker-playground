# 📄 Docker Learning — Step 1

## Запуск Node сервера в контейнере без Dockerfile

### Цель

Понять базовую модель Docker:

```
Image → Container → Process
```

И научиться запускать код внутри контейнера **без сборки собственного образа**.

---

# Минимальный Node сервер

Создали простой сервер `server.js`.

```js
const http = require("http");

const PORT = 8888;

const server = http.createServer((req, res) => {
  res.end("Hello from Docker container");
});

server.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

Проверка без Docker:

```bash
node server.js
```

Открываем в браузере:

```
http://localhost:8888
```

---

# Запуск сервера внутри Docker

Контейнер запускался командой:

```bash
docker run -it -v ${PWD}:/app -w /app -p 8888:8888 node:20 node server.js
```

---

# Разбор команды

## `node:20`

Это **образ Docker**.

Он содержит:

- Linux filesystem
- Node.js runtime
- npm
- системные библиотеки

Образ скачивается автоматически, если его нет локально.

---

## `-v ${PWD}:/app`

Bind mount.

Монтирует текущую папку в контейнер.

```
HOST
docker-playground/
└── server.js
        │
        ▼
CONTAINER
/app/server.js
```

Контейнер **не копирует файлы**, а читает их напрямую с диска.

---

## `-w /app`

Рабочая директория контейнера.

Эквивалент:

```
cd /app
```

---

## `-p 8888:8888`

Проброс портов.

```
HOST:8888 → CONTAINER:8888
```

Благодаря этому браузер может обращаться к контейнеру.

---

## `node server.js`

Процесс внутри контейнера.

Контейнер будет жить **пока работает этот процесс**.

---

# Что происходит внутри Docker

Когда выполняется команда:

```bash
docker run node:20 node server.js
```

Docker делает следующее:

1. Проверяет, есть ли образ `node:20` локально
2. Если нет — скачивает его из Docker Hub
3. Создаёт контейнер из образа
4. Создаёт writable layer
5. Запускает процесс `node server.js`

---

# Файловая система контейнера

Контейнер использует:

```
node:20 image layers
        +
container writable layer
        +
bind mount (/app)
```

Пример структуры:

```
container filesystem
│
├── /usr
├── /etc
├── /bin
├── /node
└── /app   ← mount папки проекта
```

---

# Где находится `server.js`

В текущем сценарии:

| Где       | Что                      |
| --------- | ------------------------ |
| host      | `server.js`              |
| image     | Node runtime             |
| container | процесс `node server.js` |

Файл **не хранится в образе**.

---

# Почему изменения кода применяются

Мы используем bind mount:

```
-v ${PWD}:/app
```

Поэтому контейнер читает файл напрямую с хоста.

Изменения в коде становятся видны контейнеру сразу.

Однако Node уже запущен, поэтому нужно **перезапустить контейнер**, чтобы код перечитался.

---

# Остановка контейнера

Контейнер можно остановить:

```
Ctrl + C
```

или

```bash
docker stop <container-id>
```

---

# Проверка запущенных контейнеров

```bash
docker ps
```

Показывает активные контейнеры.

---

# Ключевые выводы

1. Docker контейнер запускается из образа.
2. Контейнер — это запущенный процесс.
3. Bind mount позволяет контейнеру использовать файлы с хоста.
4. Образ не изменяется при работе контейнера.
5. Контейнер можно удалять и создавать заново.
